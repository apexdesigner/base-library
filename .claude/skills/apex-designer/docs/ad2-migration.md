# AD2 Migration

The `ad3 ad2-pull` command downloads AD2 project design data into local JSON files. These files are the input for migrating an AD2 project to AD3.

## Step 1: Pull AD2 Data

```bash
ad3 ad2-pull <project-url>
```

The URL is the AD2 project URL, e.g. `https://app.apexdesigner.io/projects/42`

This pulls the project and all its dependencies. Each re-pull overwrites all files — git tracks what changed.

### File Layout

```
ad2-data/
  source.json           ← { "url": "...", "project": "my-app", "dependencies": ["base-types", ...] }
  my-app/
    Project.<uuid>.json
    BusinessObject.<uuid>.json
    Team.<uuid>.json
    ...
  base-types/
    Project.<uuid>.json
    ...
```

- `source.json` identifies the root project and its dependencies
- Each project gets its own directory named in kebab-case
- Entity files are named `<Type>.<uuid>.json`
- Stale files and directories are automatically removed

### Entity Types Pulled

- **Project** — with dependencies, parameters, styles
- **Team** — access control roles
- **NPMDependency** — with includes (scripts, styles, assets)
- **ProjectDatasource** — database connections
- **BusinessObject** — with properties, relationships, behaviors, indexes, mixins, access rules
- **ServerFunction** — standalone functions (not attached to a business object)
- **UserInterface** — with properties, events, methods, elements (as tree), dependencies, access rules
- **Artifact** — non-generated files and project assets

### Reading the Pulled Data

- **Cross-entity references** use `uuid` strings matching filenames (e.g., `"datasource": "abc-123"` → `ProjectDatasource.abc-123.json`)
- **Intra-entity references** use names (e.g., `"onChangeMethod": "validate"` references a method by name)
- **Source code fields** (`implementation`, `stylesText`, etc.) are arrays of lines
- **Elements** are nested as a tree with `children` arrays; nested UI components use `tag` with the element selector
- **Attributes** are objects keyed by name: `{ "gap": { "expression": "0" }, "label": "Title" }` (default binding type is `value`)

## Step 2: Plan the AD3 Design Structure

### Survey the AD2 data

Use `ad3 ad2-find` to understand the project before migrating:

```bash
# Overview of all entity counts by AD3 target type
ad3 ad2-find

# Find specific target types
ad3 ad2-find business-objects
ad3 ad2-find pages
ad3 ad2-find mixins
ad3 ad2-find components
ad3 ad2-find services
ad3 ad2-find roles
ad3 ad2-find data-sources
ad3 ad2-find app-behaviors
ad3 ad2-find functions
ad3 ad2-find base-types
ad3 ad2-find artifacts
ad3 ad2-find npm-dependencies
ad3 ad2-find external-types     # skip — auto-generated by AD3
```

Also read the Project file to understand application settings, parameters, and dependencies.

### Group by ownership

Use **belongs-to relationships** as the primary grouping signal. A belongs-to relationship means ownership/containment — entities in the same ownership tree naturally belong in the same module. References are cross-module links and don't dictate grouping.

Some older AD2 projects have multiple belongs-to relationships on a single entity. Only one represents true ownership. If you encounter this, check the `onDelete` setting — a CASCADE delete typically indicates the true owner. If still ambiguous, ask the user which relationship is the real ownership one.

Build ownership trees from the belongs-to relationships, then propose modules around each root or logical cluster. For example, if `Term` owns `SelectedStudent` which owns `StudentAssignment`, those three belong in the same module.

Application-wide shared assets (base types, mixins, roles) and infrastructure server functions (e.g., Socket.IO setup, auth helpers) go in the project directory alongside project configuration.

### Create the design directory structure

Create one subdirectory per module with a markdown doc describing its contents (following the [design docs style guide](design-docs-style.md)). Create or update the root `design/design.md` as a map of all modules. Pages and components are distributed into the same module directories as their related business objects.

Use AD3 terminology in the design docs — not AD2 names. For example, use "app behavior" or "function" instead of "server function", "role" instead of "team", "page" instead of "user interface with a path."

Iterate the proposed structure with the user until they are happy with the groupings before proceeding to migration.

### Operation Access Rule Mapping

AD2 `operationAccessRules` control which CRUD operations are exposed as API endpoints and which roles can access them.

**Roles:** If all operations share the same set of teams, use `applyDefaultRoles()` on the business object with the corresponding AD3 role classes. AD2 access rules may include library teams (e.g., the Administrator team) — include the base library's `Administrator` role in `applyDefaultRoles()` alongside the project's roles.

**Exposed operations:** Only operations with `userSets: ["exposed"]` should be included in `setCrudBehaviors()`. Omitted operations are not generated as endpoints.

**AD2 operation → AD3 `setCrudBehaviors` key:**

| AD2 Operation | AD3 Key |
|---|---|
| Create | `create` |
| Find, Find By Id, Find One | `read` |
| Update By Id | `update` |
| Delete By Id | `delete` |
| Update (bulk) | `update` |
| Delete (bulk) | `delete` |

When all four CRUD operations are exposed with the same roles as `applyDefaultRoles()`, the `setCrudBehaviors()` call can use `true` for each key to inherit the default roles.

### AD3 Runtime APIs for Behavior Code

When converting AD2 behavior implementations to AD3, use these runtime APIs instead of AD2's `Model.metadata`, `this.app`, and `this.execute` patterns:

| AD2 Pattern | AD3 Equivalent |
|---|---|
| `Model.metadata.name` | `Model.entityName` |
| `Model.metadata.displayName` | `Model.entityName` (or derive from schema registry) |
| `Model.metadata.properties` | `Object.keys(Model.schema.shape)` |
| `Model.metadata.relationships` | Iterate `Model.schema.shape` entries and call `getRelationshipMetadata(field)` from `@apexdesigner/schema-tools` |
| `Model.metadata.relationships[name].businessObject.name` | `getRelationshipMetadata(field).targetEntity` |
| `Model.metadata.relationships[name].foreignKey` | `getRelationshipMetadata(field).foreignKey` |
| `Model.metadata.relationships[name].type` | `getRelationshipMetadata(field).relationshipType` (`'belongsTo'`, `'references'`, `'hasMany'`, `'hasOne'`) |
| `this.app['objects'][name]` | `App.businessObjects[name]` (import `App` from `@app`) |
| `this.execute(sql, params)` | `Model.dataSource.query(sql, params)` — returns `{ rows, rowCount }` |
| `options.request.appUser` | `await User.currentUser()` (import `User` from `@business-objects`) |
| `options` parameter on CRUD calls | Remove — AD3 uses async local storage for request context |

### Pattern Conversions

| AD2 Pattern | AD3 Pattern |
|---|---|
| Behaviors pass `options` object through call stack | AD3 uses async local storage for context |
| `supplier.save()` to persist changes | `Supplier.updateById(id, changes)` |
| Properties with `isForms` implies `<Name>FormGroup` | Use `<Name>FormGroup` directly |
| `any` used for many types | Use interface definitions and TypeScript types |

### View-Backed Business Objects

AD2 business objects with `isView: true` and `viewSQL` become view-backed BOs in AD3 using `setView()` from `@apexdesigner/dsl`.

```typescript
import { BusinessObject, setView } from "@apexdesigner/dsl";

export class SessionHistory extends BusinessObject {
  id!: number;
  studentId?: number;
}

setView(
  SessionHistory,
  `select s.id, a.student_id from attendance a join session s on s.attendance_id = a.id`
);
```

**Important:**
- View SQL must use **snake_case** table and column names (the Postgres names), not the PascalCase/camelCase entity and property names from the DSL. The persistence layer auto-converts property names to snake_case for queries, so view columns must match.
- Views are read-only — the generator omits create/update/delete methods.
- If `SCHEMA_FORCE_SYNC=true`, views may fail to create if the referenced tables don't exist yet. Workaround: temporarily disable `setView`, start the server to create the tables, then re-enable the view.
- AD2 `viewSQL` uses the original AD2 table/column naming. Translate column names to AD3's snake_case convention.

Library entities are renamed between AD2 and AD3. AD2 access rules may include library teams — include the base library's `Administrator` role in `applyDefaultRoles()` alongside the project's roles.

### Migration Workflow

1. Pull AD2 data: `ad3 ad2-pull <project-url>`
2. Survey the root project's AD2 data with `ad3 ad2-find`, group by ownership, propose a `/design` directory structure, create directories and markdown docs, and iterate with the user until agreed
3. Start with the Project entity — create the AD3 project design file
4. Migrate business objects, then server functions, then user interfaces — placing each in the agreed directory structure
5. For each entity, read the AD2 JSON and the relevant AD3 DSL doc
6. Create the AD3 design file in `/design`
7. Run `ad3 resolve` after completing a set of related edits
8. Dependency project entities don't need migration — they come from AD3 libraries
